  Changes made

  1. Removed debug print from Deltas.index (ecd.py:115, now deleted)
  The most impactful fix. index() is called constantly — inside

  alone was likely causing most of the slowness.

  Replaced the O(n) linear scan with two dicts built during infer():

  3. Avoided full deepcopy(d) in cenumerate (cenumerate, lines 309–314)
  D[i] did a full deepcopy of the Delta. The only thing that actually needed
  protecting was d.tailtypes (because cenumerate_fold calls .pop(0) on it). Now
  we access D.ds[i] directly and pass list(d.tailtypes) separately — one cheap
  list copy instead of a recursive object copy.

  4. deepcopy(tailtypes) → list(tailtypes) in cenumerate_fold (line 329)
  tailtypes is a list of type-name strings. They're immutable, so a shallow
  list() copy is all that's needed to protect against the .pop(0) in the next
  recursion.

  individual frame solutions into sols via mat_subtargets — that's exactly what

⏺ Here's the full diagnosis:

  ---
  Root cause: no assembly after explore

  mat_subtargets already registers all 28 contiguous sub-sequences of a 7-frame
  matrix as search targets. solve_enumeration finds the individual frame
  solutions quickly (budget window ~8 with 8 primitives) and stores them in
  sols. But ECD then just checks if xkey in sols and restarts, never combining 
  what it found.

  The full 7-frame solution requires ~62-node programs (budget window ~64), so
  the system burns through 8 iterations of 20 minutes each looking for what it
  already has all the pieces for.

  assemble_tconcat closes this gap: after explore it recursively tries every
  tconcat split of X, checking sols for both halves, until it either assembles
  the full solution or determines the parts aren't there yet.

  Second issue: Delta(4, int) was missing

  For a 4×4 frame, fill(0, 4, 4) is the natural base. But 4 wasn't in the DSL,
  so the system couldn't even construct a 4×4 matrix via fill. It would have had
   to find a much more tortured path (e.g. vconcat(hconcat(...)) — which you
  don't even have in D). Added Delta(4, int).

  What this means in practice

  The system should now: find all 7 frames in the first few budget windows, call
   assemble_tconcat, chain them with 6 tconcat nodes, and return — likely in the
   first ECD iteration, in seconds rather than hours.

⏺ The full chain was:

  1. saturate adds holes for each type seen in the solutions ('mat' and 'int'),
  both with repr='<>'
  2. My dict stored only the last hole's index under '<>'
  3. D[D.index('<>'):] sliced starting from the last hole — the earlier hole was
   never popped
  4. The stranded hole stayed in D.invented and got indexed into
  bytype_terminal['mat'] (it has tailtypes=None)
  5. dream's newtree could now sample the hole as a mat-type terminal
  6. hole() returns self.head = the string '<>' — 2 chars → numpy U2 dtype
  7. rep_t('<>', n) = np.tile('<>', (n, 1, 1)) → a (n, 1, 1) string array that
  passes isinstance and shape checks, then crashes tc_mat
