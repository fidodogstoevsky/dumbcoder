⏺ There are two interlocking reasons:

  ---
  1. No solutions are being found — so compression never runs

  The soltrees check at line 1132 means saturate_stitch (and therefore D.add)
  only fires when at least one solution exists. With no solutions, trees = []
  every iteration, D never grows beyond 26.

  2. Why no solutions? The new programs are too deep for the search budget

  With the old iterate-based DSL (~16 primitives), the simplest solution was:
  (iterate 4 (gset (zeros 4 4) r c 1) 1 dir)   — 11 nodes

  With 16 primitives, log-prob ≈ 11 × −2.77 = −30.5 nats → budget window 14.

  With the new HOF DSL (26 primitives), the equivalent program is:
  (unfold (gset (zeros 4 4) r c 1) 4 (step 1 dir))   — 12 nodes

  With 26 primitives, log-prob ≈ 12 × −3.26 = −39.1 nats → budget window 19.

  cenumerate sweeps budget windows [0,2], [2,4], … up to the timeout. Window 19
  (budget [38,40]) contains every possible 12-node typed tree — millions of 
  programs. At ~10k evaluations/second the enumeration doesn't reach the
  solution before the 10s timeout fires. And since idx resets to 0 at the start
  of every ECD iteration, it hits the same wall each time.

  3. Dream can't break the deadlock

  When soltrees is empty, dream(D, []) trains entirely on randomly-generated
  programs. The model learns nothing about movement patterns, so the Q it emits
  doesn't bias the search toward unfold/step_fn — it just reflects the uniform
  distribution over whatever primitives appear in random noise. This repeats
  identically for all 25 iterations.

  made each solution 1 node deeper in a DSL that's now 10 primitives wider.

❯ how might I reduce the number of int terminals?                               

⏺ The main culprits are add and mul — they're non-terminal int primitives,
  meaning every int hole can recurse into add(int, int) or mul(int, int), each
  of which opens two more int holes. That exponential branching dwarfs the
  constant terminals.

  For the current movement tasks, no arithmetic is actually needed. The base
  programs use only constant coordinates (0–3 for row/col, 4 for step count, 1
  for value). add and mul would only become useful after compression invents
  abstractions that parameterize those constants — which never happens because
  the base solutions can't be found in the first place.

  The simplest fix: remove add and mul from the initial DSL entirely. Just keep
  the int constants:

  # int terminals — no arithmetic non-terminals
  Delta(0, int, repr='0'),
  Delta(1, int, repr='1'),
  Delta(2, int, repr='2'),
  Delta(3, int, repr='3'),
  Delta(4, int, repr='4'),

  This drops D from 26 to 24, but more importantly it makes every int hole a
  dead-end leaf (depth 1, one of 5 choices) instead of a potentially unbounded
  subtree. That alone should collapse the solution from budget window ~19 down
  to something much more searchable.

  If you eventually need arithmetic (e.g. for tasks where a coordinate is
  computed rather than constant), you can add add/mul back after the first round
   of compression has already discovered the movement abstraction.
