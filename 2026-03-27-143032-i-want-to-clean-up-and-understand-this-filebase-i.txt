⏺ Here's the full flow:

  ---
  Setup (__main__)

  Build the DSL. A Deltas object D is constructed from 17 hand-written
  primitives. These are the only building blocks the system is allowed to use to
   write programs:
  - unfold — the only way to build a mat (it unrolls a grid by repeatedly
  applying a function)
  - blank (a 4×4 zero grid), gset — grid primitives
  - step_fn — constructs a fn that moves value v in a direction
  - 8 direction constants, 7 int constants (0–6)

  Build the tasks. 24 tasks total are created as (T, H, W) numpy arrays:
  - 12 simple movement tasks: a single cell 1 moving right/down/diagonally
  across a 4×4 grid over 4 frames
  - 12 navigation tasks: a 6×6 grid with walls, agent, and goal — agent takes
  BFS shortest path to goal, one step per frame

  ---
  ECD(Xs, D, timeout=120) — the main loop

  This is Enumerative Concept Discovery — it alternates between searching for
  programs and learning from what it finds.

  Iteration 0 (no model yet):
  - task_Q returns a uniform distribution over all DSL primitives (no model yet)
  - For each unsolved task, calls solve_enumeration with that uniform Q

  solve_enumeration — systematic search:
  - Enumerates complete programs from the DSL in log-probability order using
  cenumerate
  - Uses an expanding budget window: first tries programs with log-prob in [0, 
  2], then [2, 4], etc., so highest-probability programs are tried first
  - Each enumerated program is evaluated (tree()); if its output matches a
  target task matrix, it's saved as a solution
  - Stops at timeout (120s ÷ number of unsolved tasks per task)

  cenumerate / cenumerate_fold — how enumeration works:
  - Depth-first, callback-style recursive enumerator
  - Builds a full expression tree by filling argument slots one at a time,
  constrained by type (e.g. unfold needs grid, int, fn in that order)
  - The budget window gates which programs are explored

  After the search round:
  - If any solutions were found, runs saturate_stitch(D, sols) to discover
  reusable abstractions:
    - Serialises solution trees as s-expressions
    - Runs stitch_core.compress() to find repeated substructures
    - Parses each discovered abstraction, infers its argument types, and adds it
   to D as a new invented primitive
    - Returns the rewritten programs using the compressed vocabulary
  - Calls dream(D, trees) to train a MatRecognitionModel on the current solution
   trees

  dream — training the recognition model:
  - Randomly generates trees using newtree (sampling from makepaths), plus mixes
   in solution trees
  - For each tree that evaluates to a valid matrix: builds autoregressive
  training examples — predict each DSL node at position i given (the output
  matrix, all previously placed nodes)
  - Also trains an auxiliary loss: predict frame t+1 from the matrix encoder's
  hidden state at frame t
  - 100 gradient steps, returns the trained MatRecognitionModel

  MatRecognitionModel — the neural guide:
  - Takes a (B, T, H, W) int tensor (the task matrix)
  - Encodes each frame with a 2D CNN + residual, updates a GRU hidden state
  across frames
  - Separately embeds the partial program (primitives placed so far), mean-pools
   them
  - Concatenates both embeddings → linear head → logits over all DSL primitives
  - Output is a distribution over which primitive to place next

  Next iterations:
  - task_Q now uses the trained model: runs the task matrix through
  MatRecognitionModel to get a task-specific Q that biases search toward
  primitives that "look right" for this particular matrix
  - Search + compression + dreaming repeats until all tasks are solved

  ---
  Output

  Z = ECD(Xs, D, timeout=120)
  for k, v in Z.items():
      if v is not None:
          print(f'solution: {v}')         # e.g. (unfold blank 4 (step 1 right))
          print(f'evaluates to:\n{v()}')  # the (T,H,W) matrix it produces

  ---
  The core idea in one sentence

  The system tries to find short programs in the DSL that reproduce each task
  matrix, then uses the solutions it finds to (a) invent new reusable
  abstractions that make the DSL more expressive, and (b) train a neural model
  that guides future searches toward more likely programs.

❯ if cenumerate enumerates programs by probability, what does cenumerate-fold   
do?                                                                             

⏺ cenumerate and cenumerate_fold are mutually recursive and together handle two
  different sub-problems:

  - cenumerate answers: which primitive do I pick for this type slot?
  - cenumerate_fold answers: given I've picked a primitive, how do I fill in all
   its argument slots?

  Here's the flow for a single node like unfold (which takes [grid, int, fn]):

  cenumerate(type=mat, ...)
    → picks unfold
    → calls cenumerate_fold(d=unfold, tailtypes=[grid, int, fn], ...)

  cenumerate_fold sees tailtypes=[grid, int, fn]
    → pops grid, calls cenumerate(type=grid, cb=ccb)
      → ccb fires for each grid sub-tree found:
          appends it to unfold.tails
          calls cenumerate_fold(d=unfold+[grid_arg], tailtypes=[int, fn], ...)

  cenumerate_fold sees tailtypes=[int, fn]
    → pops int, calls cenumerate(type=int, cb=ccb)
      → ccb fires for each int sub-tree:
          calls cenumerate_fold(d=unfold+[grid_arg, int_arg], tailtypes=[fn],
  ...)

  cenumerate_fold sees tailtypes=[fn]
    → pops fn, calls cenumerate(type=fn, cb=ccb)
      → ccb fires for each fn sub-tree:
          calls cenumerate_fold(d=unfold+[grid_arg, int_arg, fn_arg],
  tailtypes=[], ...)

  cenumerate_fold sees tailtypes=[]
    → all slots filled — check if accumulated logp is in the budget window
    → if yes: call the original cb(complete_tree, logp)

  The "fold" name is literal — it's folding over the tailtypes list one argument
   at a time, left to right, threading the accumulated log-probability through
  each step. The budget shrinks with each argument chosen, which prunes the
  search: if you've already spent so much probability on early arguments that no
   valid completion can fall in the window, you stop early.

  So cenumerate is the "pick a node" step, and cenumerate_fold is the "fill its
  children" step — neither can work without the other.

⏺ These are the two right levers to pull. Let me break down why the navigation
  tasks are currently intractable and what would actually help.

  Why nav tasks fail today

  The DSL can't express them. The current primitives can only represent one
  thing: a single value moving in a constant direction via unfold(blank, n, 
  step_fn(v, dir)). Nav tasks require three things at once:

  1. Static layers — walls (value 3) and goal (value 2) that persist across
  frames
  2. Composition — overlaying those layers on top of the moving agent
  3. Non-constant motion — the agent follows a BFS path that turns corners, not
  a fixed direction

  The third point is the hard one. Even if you added overlay and zip_mat (which
  already exist in dsl.py but aren't in D), you still can't express "turn left
  on frame 3." unfold applies the same function at every step.

  DSL changes worth considering

  Add the HOFs that are already in dsl.py but not in D:
  - overlay (fn2) — lets the system compose the wall/goal layer with the agent
  layer
  - zip_mat — pairwise-combine two mats frame by frame
  - compose — chain two fns, enabling things like "step right then step down"
  - map_mat — apply a fn to every frame (e.g. stamp walls onto every frame)

  Add larger blank grids:
  - blank66 — nav tasks use 6×6 but blank is 4×4, so the system currently can't
  even construct the right-shaped grid

  Consider a repeat_grid or const_mat primitive — a way to say "this grid,
  repeated T times," which is exactly how you'd represent walls.

  With those additions, a nav task with a straight path becomes expressible as
  something like:
  zip_mat(overlay,
    unfold(agent_start, T, step_fn(1, right)),
    map_mat(const, walls_and_goal_grid)
  )

  The turning paths remain hard — the system would need to discover "compose
  step functions conditionally" through compression, which is unlikely without
  curriculum.

  Curriculum ideas

  The key gap is between "straight line motion" (solved) and "BFS path with
  turns" (not representable). You need intermediate tasks that bridge this:

  Stage 1 — static scenes: A grid with walls+goal but no movement. Forces the
  system to learn how to represent multi-value, multi-entity grids.

  Stage 2 — movement on larger grids: Simple motion on 6×6 so it learns blank66
  + direction combinations.

  Stage 3 — single-turn paths: Agent moves in one direction for N steps, then
  another for M steps. This is the minimum needed for corner-turning — and
  crucially, it is expressible as compose(step_fn(1, right), step_fn(1, down))
  applied via unfold if you think of it right, or as two concatenated unfolds.
  If the system can discover a "turn" abstraction here, it could apply it to nav
   tasks.

  Stage 4 — walls with straight paths: Navigation tasks where the BFS path
  happens to be a straight line (agent and goal are axis-aligned, no walls in

⏺ This is a really interesting goal. Let me think through the full abstraction
  ladder and where the architectural blockers are.

  The fundamental problem: unfold is the wrong temporal combinator

  unfold(g, n, f) applies the same f at every step. A BFS path like right, 
  right, down, right is a heterogeneous sequence — a different function at each
  step. This can't be expressed with unfold at all, regardless of what other
  primitives you add.

  You need a new primitive that builds a mat step-by-step with a different
  function at each step. Something like:

  def extend(m, f):
      "mat, fn -> mat: append one frame by applying f to the last frame"
      return np.concatenate([m, f(m[-1])[None]], axis=0)

  Then a path becomes:
  extend(extend(extend(start_mat, step_fn(1, right)),
                       step_fn(1, right)),
                step_fn(1, down))

  This is binary, no list type needed, and naturally composable. Crucially,
  stitch can compress repeated nesting of extend calls — which is the first step
   toward your abstraction ladder.

  The abstraction ladder

  Level 0 — Raw path (what the system would find first):
  zip_mat(overlay,
    extend(extend(extend(agent_start, step_fn(1,right)), step_fn(1,right)),
  step_fn(1,down)),
    repeat(walls_and_goal, T))

  Level 1 — After stitch compresses repeated structure:

  Stitch notices many programs share zip_mat(overlay, ..., repeat(scene, T)). It
   abstracts the scene composition into f0($0, $1). Also starts to see that
  extend(extend(..., step_fn(1,$0)), step_fn(1,$1)) is a recurring shape —
  abstracts it into something like move2(1, right, down).

  Level 2 — Path as a sequence:

  After enough examples, the nesting of extend + step_fn across many tasks gets
  compressed into an abstraction over direction sequences: effectively
  path(agent_val, [d1, d2, d3, ...]). This is the point you identified — "a full
   path described as a sequence of movements." Whether stitch arrives here
  depends on how many examples you give it and how uniform the program structure
   is.

  Level 3 — The jump to desires(1, 2):

  This is the hardest step and it's qualitatively different from the earlier
  ones. Levels 1–2 are syntactic compressions — stitch notices shared structure
  in the expression trees. desires(1, 2) is a semantic compression — it encodes
  why the specific directions were chosen (they're optimal for bringing 1 to 2).

  For stitch to find desires(1, 2), it would need to see many tasks where:
  - The agent value (1) and goal value (2) are the same across instances
  - The specific direction sequences differ (different paths because different
  start positions)
  - But the relationship between the directions and the agent/goal positions is
  constant

  The issue is that in the extend/step_fn representation, the agent value (1)
  appears inside each step_fn(1, dir) call, and the goal value (2) appears in
  the initial grid — but stitch has no way to notice that the choice of dir was
  motivated by where 2 is. The path directions are opaque constants from
  stitch's perspective.

  To bridge this gap, you'd probably need either:
  - A planning primitive in the DSL: bfs_path(g, from_val, to_val) → fn list —
  making the optimization explicit in the program structure, so desires becomes
  a wrapper around it
  - Or a much larger dataset so stitch sees enough variation that the agent/goal
   values become the holes and the path structure becomes the body

  What I'd actually add to the DSL

  Roughly in priority order:

  1. extend — enables heterogeneous paths at all
  2. blank66 / zeros(h, w) — nav tasks are 6×6, current blank is 4×4
  3. zip_mat + overlay — for compositing agent movement over static walls/goal
  scene
  4. A repeat_grid or single_frame primitive — to turn a static grid into a mat
  (for the walls layer)
  5. compose — already in dsl.py, add to D; useful for building more complex fns

  The higher-order mat functions (map_mat, fold_mat) can come in later rounds
  via stitch once the system has seen enough solutions.
