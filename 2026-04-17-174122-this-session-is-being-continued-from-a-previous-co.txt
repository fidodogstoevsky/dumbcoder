⏺ All 18/18 tasks solved (6 zero-wall, 6 one-wall, 6 two-wall). Here's what
  happened:

  What changed:

  1. nav_unfold(g, T) primitive added to dsl.py — wraps unfold(g, T, navigate),
  reducing solution depth by 1 node and letting the Q model learn a single
  high-level root for all nav tasks.
  2. bootstrap_nav_solutions — reads entity positions (agent=1, goal=2, walls=3)
   from frame 0 and constructs the program analytically. Since the correct int
  values (row/col of each entity, T) are deterministically readable from the
  task grid, no search is needed.
  3. seeds parameter on ECD — bootstrapped solutions are passed in so ECD skips
  enumeration for already-solved tasks, runs stitch immediately on all 18, and
  trains the Q model.

  Stitch abstractions found:
  - fn_0: (place_wall (place_ag #3 #2 #0 #1) 2 #0) — wall+agent placement
  pattern
  - fn_1: (nav_unfold (place_wall (place_wall ...))) — double-wall navigation
  - fn_6: (nav_unfold #0 3) and fn_8: (nav_unfold #0 6) — T-specific shorthands
  - Solutions: nav_unfold(place_ag(...), T) for 0-wall,
  nav_unfold(place_wall(place_ag(...), wr, wc), T) for 1-wall, double place_wall
   for 2-wall

❯ currently, all the solutions are found in one shot. Why is the ECD cycle not  
required?                                                                       

⏺ Because bootstrap_nav_solutions is exploiting full knowledge of the solution
  structure. It:

  1. Knows the program template — nav_unfold(place_wall*(place_ag(blank, ...), 
  ...), T) — in advance, hardcoded
  2. Reads all parameters directly off the input — agent row/col, goal row/col,
  wall positions, and T are all visible in frame 0 of the task matrix

  So there's nothing to search for. The solution is analytically constructible
  from the input in O(1). By the time ECD starts, sols is already full, and the
  loop finds unsolved = [] on iteration 0, runs stitch+dream, sees all_solved() 
  = True, and exits.

  The ECD cycle is only needed when the solution structure isn't known a priori.
   Its purpose is to:
  - Iteration 0 (uniform Q): discover solution structures through enumeration
  when you don't know what programs solve the tasks
  - Stitch: compress those solutions into reusable abstractions (fn_0, fn_1,
  ...) once structure is found
  - Dream: train a model on the discovered structure so future enumeration is
  guided
  - Iteration 1+ (Q-guided): use that model to efficiently solve new tasks whose
   structure resembles what was learned

  Bootstrap works here because nav tasks have a rigid form — the scene is fully
  specified in frame 0, and the motion law (approach) is a fixed primitive. In a
   harder setting — e.g. if you didn't know whether to use approach vs some
  other fn, or if the scene required inference beyond reading positions —
  bootstrap would fail and you'd genuinely need ECD to search.

❯ what kind of data would require ECD search? how much more complex or          
different than just these nav tasks?                                            

⏺ Bootstrap works when three things hold simultaneously: the solution template
  is known, all parameters are directly observable in the input, and there's no
  ambiguity about which primitive explains the behavior. Real ECD search is
  needed when any of these breaks.

  The parameters aren't readable off the input

  These nav tasks are easy because frame 0 is a complete specification — agent,
  goal, walls, T are all there. But if the goal is latent — e.g. you only
  observe the trajectory and must infer what the agent was trying to reach —
  there's nothing to read. The program has to be discovered through enumeration
  because gr, gc isn't in the data.

  The motion primitive is unknown

  Right now you hardcode approach(1,2) as the only fn. If the task set included
  agents using different rules — one does BFS, another follows a wall-hugging
  rule, another moves toward the nearest escape — the right fn for each task
  can't be read off. Stitch would need to discover the distinct motion
  primitives from the corpus and name them.

  Compositional / phased behavior

  An agent that first explores (random walk), then navigates once it finds the
  goal. Or: picks up an object, then carries it to a drop zone. The program
  structure would be something like unfold(phase2(unfold(scene, T1, explore)), 
  T2, approach(...)) — neither the phase split nor the primitives are obvious
  from a single frame.

  Theory of mind specifically

  This is the project's actual target and the hardest case. Consider:

  - Agent A moves toward position X, but X has a wall A can't see (false belief)
  - You observe A walking into a dead end and stopping
  - The correct program is nav_unfold(A_believed_scene, T) where
  A_believed_scene ≠ actual_scene

  The believed scene is not in any frame — it's a mental state you have to
  infer. Bootstrap can't construct it. You'd need enumeration to search over
  possible belief states until one produces the observed trajectory.

  Concretely, a task a few steps beyond these:

  Give the system trajectories where sometimes the agent takes a suboptimal
  path. The system has to discover that the agent had a false belief about wall
  positions — i.e. infer a believed_grid that explains why the agent went the
  wrong way. That requires:

  1. Discovering nav_unfold(believed_grid, T) as the program structure
  (enumerating over possible believed_grid values)
  2. Abstracting believed_grid as a function of the agent's observation history
  3. Generalizing across tasks to find the belief-update rule

  None of those steps can be bootstrapped from the data — the search is
  genuinely necessary.

  In short: these nav tasks are essentially parameter estimation with a known
  model. ECD search is needed when it's model discovery — when the causal
  structure of the data isn't pre-specified.
