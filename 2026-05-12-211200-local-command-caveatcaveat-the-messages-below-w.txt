⏺ Here's my read of the codebase in relation to your goal:

  ---
  What the current system has:

  The DSL is purely physical/spatial — grids, movement via BFS (approach),
  walls, temporal unfolding. No mental vocabulary whatsoever. The ECD loop can
  invent new primitives through compression across solutions.

  How mental primitives could emerge:

  The argument would go: if you give the system a corpus of tasks that humans
  would describe with mental vocabulary, compression should yield primitives
  with mental interpretations:

  - Desire → a primitive that abstracts over the (agent_val, goal_val) pattern
  in approach — the "thing the agent is moving toward" as a first-class object
  - Belief → a primitive that captures which world model drives the agent's
  trajectory (critical for false-belief tasks where the agent detours based on a
   wall it thinks is there)
  - Intention → a primitive for a planned action sequence given some
  belief/desire pair

  What seems to already be present:

  The dream model explicitly distinguishes detour vs. straight-path tasks using
  h_agent_all (agent trajectory mean) — this is essentially tracking whether the
   agent's path looks "belief-sensitive." The system is already set up to care
  about this distinction.

  What I think is missing to make the demonstration clean:

  1. A curated task corpus with clear mental structure (e.g., false-belief
  tasks: world has no wall, but agent acts as if it does)
  2. A way to evaluate whether invented primitives are semantically mental (not
  just that compression happened, but that the abstracted holes correspond to
  desire/belief slots)
  3. Possibly: tasks at multiple levels of mental nesting (1st-order belief vs.
  2nd-order)

  What specifically are you trying to build or show right now — is it the task
  corpus, the primitive evaluation, or something in the ECD loop itself?

⏺ Both issues have the same root cause: D.reset() on line 383.

  What happens:

  1. End of ECD iteration 1: stitch sees 8 nav programs → finds fn_0 = (unfold 
  #0 navigate) → registers it in D.invented
  2. Start of ECD iteration 2: saturate_stitch calls trees = [normalize(s) ...]
  first — this expands all fn_0 references back to (unfold ig_X navigate) — then
   calls D.reset(), wiping fn_0 from D
  3. Stitch now sees the full expanded corpus (28 nav + false-belief programs,
  none using fn_0) and rediscovers the same abstraction from scratch
  4. Stitch names its output starting from fn_0 every run, so the belief
  primitive found second gets fn_2 while the rediscovered nav gets a new fn_0

  The fix: serialize programs after D.reset() but using the already-normalized
  trees, then track a global counter so stitch abstractions get names that don't
   collide across iterations.

  to stitch on iteration 2, not the re-expanded originals. That way stitch sees

  uniform_type_q() assigns each grid terminal logp = -log(n_grid_terminals).

⏺ The search space is too large. The solution (unfold (gset ig_i gr gc gv) 
  (approach 1 gv)) has 5 independent int slots — gr, gc, gv in gset, plus both

  slots for the output to be correct.

  approach(1, gv). This:

  approach_1, so the discovered primitive still has the shared variable
