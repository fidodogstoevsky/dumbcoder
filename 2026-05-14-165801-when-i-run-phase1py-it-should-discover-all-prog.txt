⏺ The fix: < 0 → <= 0 at ecd.py:272.

  Why this happens: uniform_type_q assigns logp = -log(count) per type. With
  only one mat-typed symbol (unfold_auto) and one fn-typed symbol (navigate),
  their logp is -log(1) = 0. After the task-specific masking, ig_i also gets
  logp = 0. Total program cost = 0.

  The budget window for idx=0 is (0, 2). Each step adds 0 to budget[0], so it
  stays at 0 forever. The strict < 0 guard means a zero-cost program is rejected
   from every window — it literally cannot be found.

  With <= 0, zero-cost programs are correctly accepted in idx=0. Programs
  exactly at the boundary between two windows will be found twice (both windows
  have budget[0] = 0), but that's harmless since solutions deduplicates by
  matrix key.

  Bug 1 (freeze): programs = programs_compressed passes the compressed programs
  (which contain fn_0 as an opaque token) to stitch. Stitch names its new

  fn_0 inside its body. After the fn_0 → fn_1 remapping, the body becomes (mask

  expand_skipped loops forever trying to expand fn_1 using a body that contains

  programs_compressed. Stitch then sees fully expanded programs with no fn_i

  programs_compressed with programs = [str(t) for t in trees]. Now stitch always

⏺ Done. A few design decisions worth noting:

  approach_from(1) instead of approach: Using approach_from(1) as approach_1
  (type fn, takes 1 int) means the solution only has one free int variable gv in
   approach, making the shared-variable structure explicit — gv fills exactly
  one slot in gset and one slot in approach_1. With raw approach you'd have two
  ints and stitch would see the coincidence less cleanly.

  mode='agent' terminals: ig_i has the goal cell zeroed, so ECD genuinely has to
   search over (gr, gc, gv). The search space per task is roughly 4×4×10 = 160
  grid configurations times 10 for the gv in approach_1 — ~1600 programs at the
  relevant depth, very fast to enumerate.

  Reporting: the shared-variable check counts $i occurrences in the normalized
  body. If any $i appears twice, that's the desire signature. You should see
  something like:
  fn_0  [grid, int, int, int]

    *** DESIRE — shared variable $3 (×2): same value in world and behaviour ***
