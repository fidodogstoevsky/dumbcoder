  Core idea — sequential desire: Agent has an ordered pair of desires (gv1, 
  gv2). Both goals visible from the start; agent prioritizes gv1 while it
  exists, then switches to gv2.

  Solution structure:
  unfold_auto(
    gset(gset(ig, r1, c1, gv1), r2, c2, gv2),
    if_fn(exists(gv1), approach_1(gv1), approach_1(gv2))
  )

  Shared variable signatures (the "desire algebra"):
  - Simple desire: gv × 2 — "I want X" (world + behavior)
  - Sequential desire: gv1 × 3 (gset + exists + approach), gv2 × 2 — "first X,
  then Y"

  Expected stitch discoveries:
  1. fn_desire($ig, $gr, $gc, $gv) — from simple desire
  2. fn_cond_desire($gv1, $gv2) = if_fn(exists($gv1), approach_1($gv1), 
  approach_1($gv2)) — the goal algebra combinator
  3. fn_seq_desire($ig, $r1, $c1, $gv1, $r2, $c2, $gv2) — full task wrapper
  using fn_cond_desire

  Sequential tasks are harder to enumerate (19 nodes, 6 int args vs 9 nodes, 3
  int args), so they require more budget windows and the combinator emerges from
   stitch.
