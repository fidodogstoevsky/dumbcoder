  Why the hardcoding exists

  MatRecognitionModel.encode_matrix (ecd.py:1086) pools position embeddings per 
  semantic role — agent start, goal, walls, agent trajectory. To know which
  cells belong to which role, it needs the role→value mapping, and it bakes in
  the single-agent gridworld encoding convention used by the original tasks:
  0=empty, 1=agent, 2=goal, 3=wall. The values 1/2 at ecd.py:1107 and 3 at
  ecd.py:1113 are that convention frozen into the model.

  It's not arbitrary, but it is stale: the rest of the pipeline has already
  outgrown it. expand_and_compress threads an agents parameter (a list of
  (agent_value, goal_value) pairs like [(1, 2), (4, 5)]) all the way into
  dream(), and dream() itself infers agent values from agents and goal values
  from training_Xs "so nothing is hardcoded" (ecd.py:1174–1179). But at
  ecd.py:1166 it constructs MatRecognitionModel(len(D)) without passing any of
  that. The practical consequence: on a multi-agent task, cells with values 4/5
  are invisible to the recognition model — they fall into no pooling mask — so
  the Q function only ever "sees" agent 1 and goal 2.

  How to generalize

  Three options, roughly in order of how much structure you keep:

  1. Thread agents into the model (recommended). agents is already in scope at
  the construction site in dream(). Pass agent/goal/wall values into __init__,
  build the masks from those lists, and pool with a permutation-invariant
  reduction so the feature width stays 5*nembd regardless of agent count:

  def __init__(self, nd, agents=None, wall_val=3, ...):
      self.agent_vals = [av for av, _ in agents] if agents else [1]
      self.goal_vals  = [gv for _, gv in agents] if agents else [2]
      self.wall_val   = wall_val

  Then in encode_matrix, replace (x0 == 1) with a mask that's true for any agent
  value (torch.isin(x0, agent_vals_tensor) or a small loop pooling per agent
  and averaging), same for goals, and (x == self.wall_val) for walls.
  Mean-pooling across agents keeps the frame-0 start / all-frame trajectory
  structure intact — which the docstring identifies as the key signal for
  false-belief tasks — while handling any number of agents. Construction in
  dream() becomes MatRecognitionModel(len(D), agents=agents). If you want
  per-agent (rather than pooled) features, you'd instead concat one block per
  agent pair, but then mat_emb depends on agent count and the head size changes
  between task families.

  2. Infer the role→value mapping from data, the way dream() already infers
  _gv_list: values that move between frames are agents, static non-zero values
  present across frames are walls, the rest are goals. This removes even the
  agents dependency, but it's heuristic and can misclassify (e.g. a goal that
  disappears when reached looks like it "moves").

  3. Go fully value-agnostic: add a nn.Embedding(vocabsize, nembd) for cell
  values and build a DeepSets-style encoder — for each nonzero value v, pool
  positions of cells equal to v and combine with val_embed(v), summing over
  values. Maximally general (no role knowledge at all), but you lose the
  hand-built inductive biases (frame-0 start vs. all-frame trajectory mean per
  role), which that docstring says is exactly what makes detour/false-belief
  tasks distinguishable. The model would have to relearn that structure from the
  ~600-iteration dream phase, which may not be enough data.

  Option 1 is the smallest change, consistent with how the rest of the file
  already handles multi-agent generality, and it fixes the real bug (multi-agent
  cells being invisible to Q). Want me to implement it?
