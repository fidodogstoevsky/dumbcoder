⏺ This is a well-posed question, and the codebase is actually closer than it
  looks — because you went combinator-style (closures, no lambdas) in file11,
  the term machinery already handles everything; it's only the type side that's
  monomorphic. Here's the inventory of what full structure synthesis needs.

  What "full structure synthesis" means concretely

  The program must supply all three things unfold_state currently fixes:

  machine = ∃s. (grid → s) × (s → s) × (s → grid)     -- init, step, render
  mk_machine :: (grid→s), (s→s), (s→grid) → machine
  unfold_m   :: grid, int, machine → mat               -- the only interpreter
  op left

  Then "model initialized as a copy of the world" stops being interpreter code
  and becomes the program fragment init = dup, where dup :: a → pair(a,a).
  Physics synthesizes as mk_machine(id, f, id) with s = grid; belief as
  mk_machine(dup, step_program, fst) with s = pair(grid, grid); nested belief
  (an agent modeling an agent) gets s = pair(grid, pair(grid, grid)) — the
  number of channels is now an inferred type, which is exactly the property you
  want readable off solutions.

  The crucial design point: you don't need lambdas for this. Your
  on_world/on_model/sync_w are already a combinator basis specialized to one
  pair shape. The polymorphic generalization is the arrow/lens calculus —
  first(f) :: (a→a') → (pair(a,b)→pair(a',b)), second(g), fanout(f,g), comp,
  fst, snd, dup — which expresses arbitrary state-threading point-free. on_world
   is first; on_model is second. So the eager-closure evaluation in
  Delta.__call__ works unchanged, and you avoid scaling up the fragile
  _var_sentinel/_lam_impl machinery.

  The five work items

  1. Structured types + unification (~100 lines, new code, low risk). Types go
  from strings to terms: ('pair', t1, t2), ('->', t1, t2), plus type variables.
  Strings remain as nullary constructors. Standard first-order unification with
  substitutions.
  2. Thread substitutions through enumeration — the heart of it, and the
  invasive change. cenumerate/cenumerate_fold (ecd.py:185–230) currently treat a
   slot's tailtype as fixed and look up D.bytype[tp] by exact string match. With
   polymorphism, filling one slot refines sibling slots: choosing fst for a grid
   slot instantiates its argument to pair(grid, β) with β still free. So each
  enumeration node carries a substitution; candidate lookup becomes "primitives
  whose return type unifies with the requested type under the current
  substitution." D.bytype becomes an index by head constructor with a unify
  check. Same change in needle/newtree (dream's fantasy sampler) and makepaths —
   the per-type path precomputation can't survive as-is, since the candidate set
   is now substitution-dependent. This is exactly DreamCoder's enumerator, so
  there's precedent that it works with budget-windowed best-first search.
  3. Principal-type inference for stitch abstractions. typize/_annotate_holes
  infer hole types by reading the parent's tailtype — fine when types are
  ground, wrong when the body's slots contain type variables. Registering an
  invention now means running type inference over the body to get its most
  general type, then generalizing free variables. This is what makes an invented
   believe(transform, policy) reusable at any state depth — the payoff of the
  whole exercise, since a depth-2 agent constructor applied inside a depth-3
  state is how nested ToM would compress.
  4. New primitives, mostly deletions elsewhere:
  pair/fst/snd/dup/first/second/fanout/comp plus mk_machine/unfold_m. sync_w
  stays monomorphic (it's genuinely about grids). on_world/on_model/unfold_state
   become definable, hence removable.
  5. Search-space control. This is the real cost, not the code. fst's return
  type is a bare variable — it unifies with every requested type, and pair types
   nest unboundedly. You'll need a cap on type-nesting depth and you'll lean
  harder on the recognition model and curriculum. Expect the belief program to
  get a few nodes longer (explicit dup, fst render) — roughly 15–16 nodes vs
  today's 12, so the content-aware-Q budget math needs rechecking.

  The cheap experiment to run first

  Before paying for items 1–3, there's a bounded-monomorphization shortcut:
  mechanically instantiate the pair calculus at a fixed family of state types
  (grid, pair(grid,grid), maybe one deeper) — i.e., generate monomorphic copies
  dup_g, fst_gg, second_gg, … as ordinary string-typed primitives, plus
  per-shape mk_machine/unfold_m. Zero enumerator changes; you find out this week
   whether search can discover dup + second-channel routing when nothing
  privileges it. The tradeoff is that abstractions don't transfer across state
  shapes (stitch sees fst_gg and fst_g_gg as unrelated tokens), so it can't
  deliver the nested-belief payoff — it's a de-risking probe, not the
  destination.

  My recommendation: run the monomorphic probe first; if search reaches
  mk_machine(dup, …, fst_gg) on your belief corpus, the science is validated and
   the unification enumerator (the one genuinely hard piece) is worth building.
